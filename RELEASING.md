# Releasing Ralph

> This document is the single source of truth for how Ralph's three artifacts are
> released. It supersedes `docs/PUBLISHING.md`.

Ralph ships three independently versioned components:

| Component       | Path                      | Artifact                       | Tag schema             |
| --------------- | ------------------------- | ------------------------------ | ---------------------- |
| `ralph-core`    | `packages/core`           | npm `@daonhan/ralph-core`      | `ralph-core-vX.Y.Z`    |
| `ralph`         | `apps/cli`                | npm `@daonhan/ralph`           | `ralph-vX.Y.Z`         |
| `ralph-sandbox` | `packages/core/templates` | Docker `daonhan/ralph-sandbox` | `ralph-sandbox-vX.Y.Z` |

Releases are driven by [release-please](https://github.com/googleapis/release-please)
in manifest mode. The config lives at the repo root in `release-please-config.json`
and `.release-please-manifest.json` (the manifest is the source of truth for the
last-released version of each component).

## Version policy

- **Independent versions per component.** The three components are never released
  in lockstep; a change to one does not force a bump of the others.
- **0.x is treated as a stable contract.** This is stricter than the SemVer spec's
  "0.x means anything goes" allowance: while on 0.x, a backward-compatible feature
  is a **minor** bump, a bug fix is a **patch** bump, and any **breaking change
  forces `1.0.0`**. Downstream users can pin a 0.x version and trust that a patch
  or minor upgrade will not break them.
- Pre-release tags (`-rc.N`, `-next.N`) are opt-in per release via a `release-as:`
  commit footer (documented in a later phase).
